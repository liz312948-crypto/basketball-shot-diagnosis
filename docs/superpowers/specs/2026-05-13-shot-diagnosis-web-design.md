# 投篮诊断 Web Demo 设计

## 1. 目标

本项目第一版要解决的不是“做一个篮球网站”，而是“让投篮分析比现有原型明显更准，并且让普通用户能顺手用起来”。

第一版交付物定义为：

- 一个可用的 Web 上传与结果查看界面
- 一个分层的投篮分析流水线
- 一个能显式表达可信度的诊断报告
- 一个能输出标注视频与关键问题解释的结果页

核心成功标准：

- 用户能完成 `上传 -> 等待 -> 查看结果`
- 系统能区分高质量与低质量视频，避免对差视频给出自信废话
- 高质量视频能触发增强诊断，结果明显强于当前启发式原型
- 结果页先让普通用户看懂，再允许进阶用户继续深看


## 2. 用户与使用场景

首要目标用户：

- 普通篮球爱好者

次级兼容用户：

- 认真训练的球员

这意味着产品表达必须遵守两个原则：

- 首页、上传、结果摘要必须低认知负担，避免专业术语堆砌
- 深层结果仍要保留分阶段动作分析与训练建议，方便进阶用户展开查看

第一版支持的视频场景：

- 短视频单次投篮
- 单回合片段
- 更长训练视频

但诊断优先级必须明确：

- 主路径优先服务短视频与单回合片段
- 长视频可接收，但只在检测到可分析出手片段时提供完整报告
- 对低质量或复杂长视频要明确降级，而不是硬做满血诊断


## 3. 产品定位

产品定位不是完整商业化网站，而是：

`精度优先的可用 Web demo`

这一定义带来三个直接约束：

- 不做账号体系与历史中心
- 不追求平台化功能
- 工程重心放在分析准确率、结果可信度、可视化解释

用户价值主张：

- 上传投篮视频
- 系统判断这段视频是否适合分析
- 系统输出可信度清晰、结论清楚、训练建议可执行的诊断结果


## 4. 第一版范围

### 做

- Web 上传页面
- 视频基础校验
- 预检与质量分级
- 基础诊断
- 高质量视频触发增强诊断
- 可信度显式展示
- 混合型结果页
- 标注视频输出
- 理想模板对比

### 不做

- 账号体系
- 历史报告列表
- 社交分享链路
- 教练后台
- 多球员同时分析
- 球星模板库
- 实时摄像头分析
- 原生手机 App


## 5. 现有原型问题

现有 `basketball_clip_tool` 原型可复用的部分主要是：

- 视频上传与本地保存
- 基础事件检测与裁剪
- 一部分视频输出流程

现有实现的主要缺陷：

- 同步请求直接跑完整分析，不适合长任务
- 分析逻辑单层、规则简单，上限低
- 缺少机位与视频质量预检
- 缺少显式可信度体系
- 结果页只是文件输出，不是产品化报告
- 没有“基础诊断 / 增强诊断”的结构边界

因此第一版不是“页面翻新”，而是“重构续建”：

- 尽量复用可用的视频处理组件
- 重写面向产品的任务流、分析流与报告流


## 6. 总体架构

系统拆为五层：

1. 上传与任务层
2. 预检层
3. 基础诊断层
4. 增强诊断层
5. 报告层

### 6.1 上传与任务层

职责：

- 接收视频上传
- 校验文件类型、大小、时长
- 创建任务 ID
- 触发后台分析任务
- 返回任务状态给前端

设计要求：

- 上传成功后不要保持同步阻塞直至分析完成
- 分析过程必须有明确状态，例如 `uploaded / precheck / base-analysis / enhanced-analysis / rendering / done / failed`
- 匿名使用即可，不建立用户账户

### 6.2 预检层

职责：

- 判断机位类型：正面 / 侧面 / 45 度 / 混乱
- 判断清晰度、稳定性、遮挡、主体完整度
- 判断是否存在足够完整的出手窗口
- 输出 `analysis eligibility score`

设计要求：

- 预检结果必须进入最终报告
- 差视频不能假装与好视频同等可信
- 预检失败时仍可提供重拍建议或有限反馈

### 6.3 基础诊断层

职责：

- 对可分析视频输出基础结论
- 基于姿态、球轨迹、身体稳定性、动作节奏识别主要问题
- 输出普通用户可理解的摘要与训练建议

设计要求：

- 所有进入诊断的视频都尽量先经过这一层
- 即便未触发增强诊断，也要能产出最低可用结果
- 输出必须结构化，供报告层直接消费

### 6.4 增强诊断层

职责：

- 仅对高质量视频触发更强模型
- 提供更细的动作阶段分析
- 提供与理想模板的偏差说明
- 生成更细训练建议与解释文本

设计要求：

- 必须与基础诊断解耦，避免强模型变成唯一依赖
- 允许后续替换或新增模型，而不改动上层产品结构
- 结果必须带来源等级，便于在报告中区分“基础结论”和“增强结论”

### 6.5 报告层

职责：

- 合并预检、基础诊断、增强诊断结果
- 生成最终报告数据
- 生成标注视频与关键帧
- 驱动前端结果页展示

设计要求：

- 结果先给总体结论，再给分层展开
- 可信度必须显式可见
- 低质量视频必须解释“为什么这次结论可信度较低”


## 7. 分层分析流水线

流水线采用三段式：

1. 预检
2. 基础诊断
3. 增强诊断

### 7.1 预检

输入：

- 原始视频

输出：

- 机位类型
- 视频质量评分
- 遮挡与完整度判断
- 可分析性评分
- 是否允许进入增强诊断

决策规则：

- 低质量视频：只给基础反馈或重拍建议
- 中质量视频：给基础诊断，不触发增强模型
- 高质量视频：给基础诊断，并触发增强诊断

### 7.2 基础诊断

目标：

- 先给稳定、快速、可解释的结果

输出：

- 一句总体判断
- 3-5 个主要问题
- 每个问题对应训练建议
- 基础可信度说明

主要特征来源：

- 人体姿态关键点
- 球轨迹
- 重心稳定性
- 动作节奏
- 机位与画面质量

### 7.3 增强诊断

目标：

- 对优质视频做更深入、更像专业分析的结果增强

输出：

- 准备阶段分析
- 下蹲阶段分析
- 起跳阶段分析
- 出手阶段分析
- 跟随动作分析
- 理想模板偏差说明
- 更细的训练建议

增强策略：

- 基础视觉/规则层负责稳定骨架
- 更强模型负责补充更精细的动作理解与报告表达

这样做的理由：

- 差视频不浪费高成本模型
- 强模型只作为增强器，不接管整条链路
- 出问题时可以定位是预检、基础分析还是增强层失真


## 8. 可信度体系

可信度必须作为一等结果，而不是隐藏字段。

第一版定义三个等级：

- 高
- 中
- 低

可信度组成因素：

- 机位适配度
- 画面清晰度
- 人体关键点完整度
- 球轨迹可追踪性
- 遮挡程度
- 出手窗口完整度
- 是否触发增强诊断

展示原则：

- 总报告顶部直接显示可信度等级
- 同时显示形成该等级的简短原因
- 可信度低时，优先展示重拍建议

示例表达：

- `高可信度：机位稳定，出手与球轨迹完整，已触发增强诊断`
- `中可信度：主体可见，但球轨迹有局部丢失，未触发增强诊断`
- `低可信度：机位混乱且出手阶段遮挡明显，本次仅提供基础反馈`


## 9. 结果页设计

结果页采用“混合型报告”。

### 9.1 顶部摘要

展示：

- 总体判断
- 可信度等级
- 是否触发增强诊断
- 视频适配度简述

目标：

- 普通用户进入页面 5 秒内知道这次结果值不值得信

### 9.2 主要问题

展示：

- 3-5 个最重要问题
- 每个问题配一句简短解释
- 每个问题配一个训练建议入口

目标：

- 保证非专业用户先看懂“我哪里有问题”

### 9.3 动作阶段拆解

展示：

- 准备
- 下蹲
- 起跳
- 出手
- 跟随动作

目标：

- 给认真训练用户更细拆解

交互原则：

- 默认折叠或半展开，避免首页信息爆炸

### 9.4 训练建议

展示：

- 2-4 条按优先级排序的建议

设计要求：

- 必须可执行
- 不堆砌术语
- 不给十几条碎建议

### 9.5 可视化结果

展示：

- 标注视频
- 关键帧截图
- 关键点、角度、轨迹、动作提示

目标：

- 让用户看到“系统为什么这么判断”

### 9.6 重拍建议

触发条件：

- 可信度中低
- 或预检已识别出明显拍摄问题

展示：

- 建议拍摄角度
- 建议距离
- 建议保持完整入镜
- 建议时长或机位稳定方式


## 10. 首页与任务状态页

### 10.1 首页

首页只做一条主路径：

- 上传视频
- 说明推荐拍摄方式
- 说明支持的视频范围

首页必须传达的内容：

- 支持短视频、单回合片段、较长训练视频
- 高质量侧面或 45 度视频能得到更准结果
- 差视频会被降级分析

### 10.2 分析中状态页

必须显示明确进度阶段：

- 视频上传完成
- 预检中
- 基础诊断中
- 增强诊断中（如适用）
- 生成标注视频
- 生成报告

设计目标：

- 降低等待焦虑
- 明确系统不是卡死，而是在逐步分析


## 11. 视频与任务处理策略

第一版采用云端处理，匿名使用。

任务处理原则：

- 上传后创建任务，不要求登录
- 视频用于本次分析
- 可以保留短期文件用于结果查看，但不设计长期历史库

第一版应考虑的任务属性：

- 上传时间
- 原始文件元信息
- 预检结果
- 基础诊断结果
- 增强诊断结果
- 生成产物路径
- 当前状态
- 错误信息

即使第一版先用轻量本地任务队列，也要保留任务对象结构，避免以后再拆。


## 12. 技术设计原则

### 12.1 组件边界

建议最少拆成以下模块：

- `ingest`
- `precheck`
- `base_analysis`
- `enhanced_analysis`
- `reporting`
- `rendering`
- `jobs`
- `web`

每个模块只做一件事，避免把“读视频、跑模型、拼文案、渲染页面”写进一个文件。

### 12.2 可替换的增强模型接口

增强层必须抽象成统一接口，例如：

- 输入：视频片段、预检数据、基础分析结果
- 输出：阶段性诊断、模板偏差、增强建议、可视化标记数据

这样后续可以替换不同模型，而不影响整站逻辑。

### 12.3 降级策略

系统必须允许这些场景：

- 预检失败但仍生成重拍建议
- 基础诊断成功但增强诊断跳过
- 标注视频失败但文本报告仍可返回

不能因为某一层失败就把整个结果页打成 500。


## 13. 错误处理

需要覆盖的错误类型：

- 非法文件格式
- 文件过大或时长超限
- 视频无法解码
- 未检测到可分析投篮
- 预检分数过低
- 基础诊断异常
- 增强模型调用失败
- 标注视频生成失败

处理原则：

- 用户总能看到可理解的结果状态
- 错误要转译成用户能懂的话
- 如果能部分成功，就返回部分结果


## 14. 测试策略

测试分三层：

### 14.1 单元测试

- 预检评分逻辑
- 可信度计算
- 诊断结果合并
- 降级与容错逻辑

### 14.2 流程测试

- 上传短视频并完成整条链路
- 中质量视频只走基础诊断
- 高质量视频触发增强诊断
- 低质量视频返回降级结果

### 14.3 样本验证

- 准备一组不同机位、清晰度、时长的视频样本
- 对比现有原型与新方案结果
- 确认新系统在可信度表达和问题判断上明显更像“可靠工具”而不是“随机点评器”


## 15. 第一版交付判断标准

第一版算成功，不看页面数量，看以下问题是否成立：

- 用户能顺利上传并拿到结果
- 系统能区分哪些视频值得深度分析
- 结果报告能清晰解释为什么可信或不可信
- 高质量视频的结果明显强于旧原型
- 标注视频与文字结论能相互印证

如果做不到这些，页面再漂亮也只是空壳。


## 16. 后续扩展方向

不纳入第一版，但架构上要预留：

- 账号与历史报告
- 教练视角报告
- 球星模板库
- 多次投篮一致性分析
- 批量训练 session 分析
- 手机端采集与上传优化
